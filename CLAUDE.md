# CLAUDE.md

Инструкции для Claude Code в этом репозитории. Читаются автоматически при
старте. Держать коротким и актуальным.


## Проект

R&D-бенчмарк LLM-провайдеров для модерации пользовательских отзывов
(текст + фото) на маркетплейсе туристических активностей. Стратегия и
полная картина — в `ARCHITECTURE.md`. Таксономия правил — в `rules.yaml`
(единственный источник правды).


## Стек

- Node.js 20+, TypeScript, pnpm
- Promptfoo как движок бенчмарка
- dotenv для секретов
- sharp для синтетических модификаций изображений

Всегда `pnpm`, никогда `npm` или `yarn`. Если в PR видишь `package-lock.json`
— это ошибка, удалить.


## Ключевые файлы

- `rules.yaml` — таксономия из 35 текстовых + 15 фото-правил
- `configs/*.yaml` — Promptfoo-конфиги (по одному на задачу)
- `providers/*.ts` — custom providers для Promptfoo
- `prompts/*-vN.txt` — версионированные промпты
- `datasets/` — тестовые данные
- `reports/` — генерируемые отчёты прогонов
- `.env.example` — шаблон переменных, `.env` gitignored


## Провайдеры и ключи

Активны: Yandex (YANDEX_API_KEY + YANDEX_FOLDER_ID), GigaChat (GIGACHAT_AUTH_KEY
+ GIGACHAT_SCOPE).
Qwen отложен — не пытайся добавить DashScope / OpenRouter без запроса.

GigaChat OAuth на `ngw.devices.sberbank.ru` требует доверия к корневому
сертификату Минцифры — через `NODE_EXTRA_CA_CERTS`. Если `pnpm smoke` даёт
`fetch failed` на GigaChat — это всегда TLS, не ретрай и не трогай код.
Инструкция и шаблон `.claude/settings.local.json.example` — в
`providers/README.md`, раздел «TLS: сертификат Минцифры».


## Конвенции кода

1. **TypeScript строгий.** В `tsconfig.json` включены `strict`, `noImplicitAny`,
   `noUncheckedIndexedAccess`. Не отключай ради скорости.
2. **Идентификаторы в коде — английские.** Данные, промпты, сообщения
   пользователю — русские.
3. **Температура по умолчанию 0.** Любое отклонение — отдельным коммитом с
   обоснованием в описании.
4. **JSON-выход моделей строгий.** В промптах явно указывать формат; в
   ассершенах использовать `is-json` + кастомную JS-валидацию схемы.
5. **Кэш Promptfoo включён.** Не сбрасывать без необходимости. Ключ кэша
   должен меняться при смене промпта или модели, но не при смене unrelated
   параметров прогона.
6. **Один провайдер = один файл в `providers/`.** Не смешивать клиентов и не
   делать общий супер-клиент.
7. **OAuth-токены кэшируются** с учётом TTL (GigaChat 30 минут, обновляем за
   2 минуты до истечения), с single-flight-мьютексом на refresh. Никаких
   запросов токена на каждый вызов и никаких гонок на параллельном обновлении.
8. **Retry-цикл HTTP-клиента ретраит не только 429/503**, но и сетевые ошибки
   (`ECONNRESET`, `ETIMEDOUT`, undici `TypeError: fetch failed`, `AbortError`
   без пользовательского `AbortSignal`). Без этого длинные прогоны теряют
   случайные кейсы на обрывах соединения. Пользовательский `abortSignal`
   (Ctrl+C в Promptfoo) уважается — без ретрая.
9. **Модель провайдера задаётся явно.** У `GigaChatProvider` дефолта нет: без
   `config.model` конструктор бросает. Это защита от «случайно погнали всё
   через самую дорогую модель». В YAML-конфигах Promptfoo модель всегда
   указывать явно.
10. **Тарифы провайдера — в коде**, не в env. Таблица `DEFAULT_PRICING` в
    `providers/<name>.ts`, override — через `config.pricing` в YAML.
    `metadata.pricingSource` в ответе провайдера говорит, откуда пришла
    цена (`config` / `default-table` / `merged` / `none`). Перед публикацией
    отчёта сверять таблицу с актуальным прайсом провайдера.
11. **Кэш загруженных в `/files` картинок — на диске.** `.cache/<provider>-files.json`,
    TTL 24 часа (GigaChat удаляет файлы через ~48ч, берём половину с запасом).
    Между прогонами `promptfoo eval` идентичные картинки не перезаливаются.
    `.cache/` — в `.gitignore`.
12. **Логи не должны содержать секретов и полный текст отзывов.** При дебаге —
    обрезать до 200 символов и хэшировать ключи заказов.


## Правила работы с промптами

- Новая итерация → новый файл: `text-moderation-v1.txt` → `text-moderation-v2.txt`.
  Никогда не перезаписывай существующую версию.
- В шапке новой версии пиши краткую diff-заметку: что изменилось и зачем.
- В конфигах Promptfoo разные версии промптов подключаются как параллельные
  `tests` — чтобы в одном прогоне увидеть эффект.


## Правила работы с `rules.yaml`

- Никогда не редактировать молча. Если нужно добавить/изменить правило —
  спросить у пользователя и получить явное подтверждение.
- Поля `show_to_organizer` и `organizer_message` исключены умышленно. Не
  возвращать.
- При добавлении правила: соблюдать схему (id, category, title, description,
  example, severity), уникальность id, соответствие формата `RVW-TXT-NN` или
  `RVW-IMG-NN`.


## Правила работы с датасетами

- Сгенерированные синтетические данные всегда пишутся в `datasets/` с
  метаданными (`source: synthetic`, какая модель сгенерировала, какие `rule_id`
  целевые).
- Реальные данные (если появятся) лежат отдельно с пометкой `source: production`
  и очищаются от персональных данных перед коммитом.
- Фото с сомнительным содержимым (насилие, дети, откровенный контент) —
  **не генерировать синтетически и не скачивать**. Только из реальной выгрузки
  с согласования.


## Эксперименты

Одна переменная за прогон. Если хочется поменять сразу три вещи — разбей на
три коммита и три прогона.

Первый прогон новой конфигурации — **всегда на подмножестве** (10–20 кейсов,
через `numTests` в конфиге), чтобы убедиться, что пайплайн собран. Полный
прогон — после того как маленький прошёл чисто.

Перед финальными выводами — 3 повторных прогона для проверки стабильности.


## Команды

```bash
# установка
pnpm install

# прогон одного конфига
npx promptfoo eval -c configs/text-moderation.yaml

# прогон на подмножестве
npx promptfoo eval -c configs/text-moderation.yaml --numTests 10

# несколько повторов для стабильности
npx promptfoo eval -c configs/text-moderation.yaml --repeat 3

# посмотреть результат
npx promptfoo view
```


## Hard rules — чего не делать никогда

- Не коммитить `.env`, ключи, токены ни в каком виде.
- Не вставлять реальные ключи в примеры, комментарии, тесты, фикстуры.
- Не отправлять запросы к Qwen / DashScope / OpenRouter пока пользователь не
  разблокировал направление явно.
- Не скачивать произвольные модели или веса — работа только через API.
- Не «чинить» `rules.yaml` автоматически, даже если что-то кажется опечаткой.
- Не заменять Promptfoo на свою самописную обвязку «потому что проще».
