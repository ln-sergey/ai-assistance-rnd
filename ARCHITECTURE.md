# Architecture — AI Bench для модерации отзывов

## 1. Цель и контекст

R&D-проект: сравнительный бенчмарк LLM-провайдеров для автоматической
модерации пользовательского контента на маркетплейсе туристических активностей.
Результат — обоснованный выбор провайдера (или их комбинации) для будущей
интеграции в прод, с компромиссом между качеством, стоимостью и латентностью.

Задача полностью R&D. Продовая интеграция, SLA, мониторинг и бизнес-процессы
обработки отклонённых отзывов — вне периметра этой работы.


## 2. Scope

### In scope

- **Модерация текста отзывов** по 35 правилам (`rules.yaml:text_rules`)
- **Модерация фотографий отзывов** по 15 правилам (`rules.yaml:image_rules`)
- Сравнение **двух провайдеров**: YandexGPT / Yandex Vision и GigaChat / GigaChat Vision

### Out of scope

- **Qwen** — отложен до решения вопроса с оплатой из РФ. Архитектура должна
  позволять подключить его без переписывания ядра: отдельный файл провайдера,
  отдельная строка в матрице прогона.
- **Чат техподдержки** — методика оценки RAG-задач (faithfulness, answer
  relevance, citation accuracy) принципиально другая. Если потребуется — будет
  отдельный бенчмарк, возможно на победителе этого.
- **Улучшение и генерация описаний активностей** — был в исходном плане,
  вынесен из текущей итерации.
- **Fine-tuning, эмбеддинги, классический ML** — рассматриваются только LLM из
  коробки.
- **Бизнес-поля `show_to_organizer` / `organizer_message`** — логика
  продовой подачи причин, не имеет отношения к R&D.


## 3. Оцениваемые провайдеры и модели

| Провайдер     | Текст                                     | Картинки                 | Статус   |
|---------------|-------------------------------------------|--------------------------|----------|
| Yandex Cloud  | YandexGPT Lite, YandexGPT, YandexGPT Pro  | Yandex Vision + YandexGPT| активно  |
| Sber          | GigaChat Lite, GigaChat, GigaChat Max     | GigaChat Vision          | активно  |
| Alibaba       | Qwen 2.5 / Qwen 3 серия                   | Qwen VL                  | отложен  |

Конкретный список версий моделей фиксируется на старте прогона и пишется в
отчёт — Yandex и Sber обновляют линейку регулярно, результат валиден только
для указанных версий.

Для картинок у Yandex отдельный шаг: Vision делает предобработку (OCR,
детекция объектов, оценка технического качества), её результат склеивается с
YandexGPT для семантической классификации по `image_rules`. У Sber GigaChat
Vision — прямая multimodal-классификация одним вызовом. Разница архитектур
сама по себе часть бенчмарка: пайплайн из двух вызовов против одного.


## 4. Стек и инструменты

- **Node.js 20+**, TypeScript для custom-провайдеров
- **pnpm** как пакетный менеджер
- **Promptfoo** — движок прогона: веб-UI для результатов, custom providers,
  llm-judge, параметризованные конфиги, агрессивный кэш по
  `(provider, prompt, input)`
- **dotenv** для секретов
- **sharp** для синтетической модификации изображений
- **git** — единственный источник правды. В него синхронизируются и Chat
  (через Project knowledge), и Claude Code (рабочая копия репо).

## 6. Формат входа и выхода моделей

Вход — отзыв (текст и/или URL/base64 изображения) плюс таксономия правил,
инжектируемая в системный промпт из `rules.yaml`.

Выход строго JSON:

```json
{
  "violations": [
    {
      "rule_id": "RVW-TXT-10",
      "severity": "critical",
      "quote": "пишите мне на +7 900 123-45-67",
      "confidence": 0.92
    }
  ],
  "verdict": "reject"
}
```

`verdict` выводится из списка нарушений: есть хотя бы одно `critical` или
`high` → `reject`; только `medium` → `needs_review`; иначе `approve`. Это
детерминистичная логика, которая считается на стороне ассершенов Promptfoo
— модели не поручается, чтобы отделить ошибку классификации правил от
ошибки принятия решения.

`quote` обязателен для текстовых правил и для правил на картинках с
идентифицируемым текстовым фрагментом (надпись, подпись, OCR-текст): это
антигаллюцинационный якорь — правило заявлено, но такого фрагмента в исходнике
нет → считаем нарушение ложным.

Температура по умолчанию `0`. Если модель с нулём нестабильна (видели на
некоторых ранних GigaChat), фиксируется `0.1` с явным обоснованием в README
соответствующего конфига.


## 7. Методика оценки

### 7.1. Метрики качества

**Текстовая модерация:**

- Precision / recall / F1 по каждому `rule_id` отдельно и усреднённые (macro)
  по всему набору
- Precision / recall / F1 в разрезе severity-групп — критические правила
  важнее, чем low
- Hallucination rate — доля случаев, когда заявленного `quote` в исходном
  тексте нет
- Verdict accuracy — совпадение `approve / needs_review / reject` с разметкой

**Модерация картинок** — те же метрики, плюс:

- Two-stage vs one-stage effectiveness — сравнение пайплайна «Vision + GPT» с
  «GigaChat Vision» по идентичным кейсам

### 7.2. Метрики ресурсов

- Cost per 1000 отзывов — по токенам × тарифу провайдера на момент прогона;
  для картинок — плюс стоимость Vision-вызовов
- Latency p50 / p95 — на реальных вызовах, не на кэшированных
- Token utilization — средняя длина входа и выхода для прикидок квот

### 7.3. Стабильность

Финальные конфиги прогоняются **3 раза** подряд с идентичными параметрами.
Считается:

- Agreement rate — доля кейсов, где множество `rule_id` совпало ровно между
  всеми тремя прогонами
- Verdict agreement — то же для итогового `verdict`

Модель с verdict agreement < 85 % снимается с рассмотрения вне зависимости от
средней F1. Порог эмпирический и будет уточнён после первых прогонов.


## 8. Датасеты

Три источника, по возможности сочетать:

1. **Реальные отклонённые отзывы** из истории модерации (если выгрузка
   возможна) — самый ценный материал. Потребует разметки: Claude Code матчит
   свободные комментарии модераторов с `rule_id` из `rules.yaml`, человек
   просматривает только `needs_review` записи.
2. **Реальные одобренные отзывы** — негативные кейсы для precision; ловят
   модели, которые «флагают всё подряд».
3. **Синтетические кейсы** — генерируются Claude Code по списку `rule_id`.
   Каждый кейс: один-два целевых `rule_id`, остальной контент нейтральный.
   Отдельно — **нарушения с нюансом** (граничные случаи): пересечения правил,
   сарказм, завуалированные формулировки.

Для картинок — то же, плюс **технические синтетические дефекты**: исходные
нормальные фото пропускаются через `sharp` с преобразованиями (blur, dark,
watermark, collage, наложение контактов, имитация афиши, имитация скриншота).
Семантические нарушения (насилие, алкоголь, дети) синтетически не
генерируются по этическим соображениям — только из реальных отклонений либо
честно выводятся из бенчмарка с оговоркой.

Целевой размер:

- Текст: 200–400 кейсов с перевесом в граничные
- Картинки: 80–150 кейсов


## 9. Жизненный цикл промптов

Промпты **версионируются файлами**, не редактируются на месте.

- `prompts/text-moderation-v1.txt` — стартовая версия
- `prompts/text-moderation-v2.txt` — после итерации, с diff-заметкой в шапке:
  что изменилось и зачем

В конфиге Promptfoo перечисляются нужные версии как параллельные тест-сьюты —
так одна модель прогоняется на разных промптах, и видно, что именно дал
рефакторинг.

**Правило экспериментов:** за раз меняется одна переменная (промпт ИЛИ модель
ИЛИ температура ИЛИ датасет). Параллельное кручение нескольких ручек — главный
источник ложных выводов в бенчмарках.


## 10. Критерии финального выбора

Веса фиксируются **до** прогонов, чтобы исключить подгонку:

- **Пороги отсечения** (hard gates):
  - precision по critical-правилам ≥ 0.90
  - recall по critical-правилам ≥ 0.85
  - verdict agreement ≥ 0.85
- **Основные факторы** после отсечения:
  - macro-F1 по всей таксономии
  - cost per 1000 отзывов
  - latency p95
- **Дополнительно:**
  - стабильность JSON-выхода (% валидных по схеме с первой попытки)
  - удобство batch-отправки, если есть

Итог — Markdown-матрица в `reports/summary.md` по завершении полного прогона.


## 11. Риски и ограничения

- **Модели подвижны во времени.** Yandex и Sber обновляют линейку; результат
  валиден на момент прогона. Версии фиксируются в отчёте.
- **Малые датасеты → широкие доверительные интервалы.** С 200 кейсами разница
  в 2–3 % между моделями чаще шум, чем сигнал. Значимой считается разница от
  5 % по F1.
- **Субъективность разметки картинок** по ряду правил («вводящее в
  заблуждение фото», «подрывает доверие к сервису»). Ожидаемое межразметочное
  согласие 0.7–0.85, не выше.
- **OCR в Yandex Vision** не идеален для рукописного и афишного текста. Это
  даёт фору GigaChat Vision на соответствующих кейсах — учесть при
  интерпретации.
- **Региональный риск провайдеров.** Блокировка или изменение условий доступа
  к одному из сервисов на стороне провайдера — не исключение; архитектура
  должна позволять легко снять провайдера с прогона.


## 12. Рабочий контур

- **Claude Chat, этот Project** — планирование, промпт-инжиниринг, разбор
  результатов, написание rubric'ов, поддержка этого документа
- **Claude Code** — скаффолдинг, код провайдеров, синтез датасетов, прогон
  Promptfoo, генерация отчётов
- **Git-репозиторий** — мост и единственный источник правды

Project knowledge держится в актуальном состоянии: при значимых изменениях
`rules.yaml` или `ARCHITECTURE.md` — перезалить файлы в knowledge текущего
Project'а.
