# ТЗ — ручная валидация baseline + разметка реальных карточек (Sprint P3)

Следующая сессия AI-агента. Документ самодостаточен: читается «с холода»,
предполагает только знание `AGENTS.md`, `ARCHITECTURE.md`,
`text_rules.yaml` + `image_rules.yaml`, двух JSON Schema в
`datasets/schema/` и результатов спринтов P1–P2 (см. `docs/tz-parsing.md`).

## Цель

Пройтись по **67 реальным карточкам**, собранным в P1–P2, и вручную
(силами агента) для каждой:

1. Проверить, что поля заполнены осмысленно, а не парсерным мусором.
2. Мысленно прогнать её через **все 35 текстовых правил**
   `text_rules.yaml` (TXT-01…TXT-35).
3. Классифицировать: «чистая» (нарушений нет) или «грязная» (есть ≥ 1).
4. Для «грязных» — зафиксировать `expected_violations` с точным
   `field_path` и `quote`; для «чистых» — `expected_clean: true`.

Результат — **baseline реальных тест-кейсов** для бенчмарка модерации
карточек (Promptfoo-прогоны) + **список правил, у которых реальных
примеров не нашлось** — эти правила поедут в следующий спринт через
синтетику.

## Scope / Out of scope

**In scope:**

- Классификация 67 карточек из `datasets/*/cards.raw.jsonl` (итог P2).
- Генерация файлов кейсов в `datasets/cases/real-clean/` и
  `datasets/cases/real-dirty/`, валидных против
  `datasets/schema/test_case.schema.json` (вариант `card_case`).
- Аудит-отчёт `docs/real-cards-audit.md` с распределением найденных
  нарушений по rule_id / severity.
- Список правил без реальных примеров → кандидаты на синтетику.

**Out of scope (явное решение пользователя):**

- Разметка нарушений фото-правил (IMG-01…IMG-30) — отдельный спринт.
- Генерация синтетических кейсов — отдельный спринт.
- Изменение `text_rules.yaml` / `image_rules.yaml` в любом виде.
- Редактирование исходных `cards.raw.jsonl` — они иммутабельный вход;
  если в карточке мусор, фиксируем это в `notes`, не «чиним» парсер.

## Вход

Продукт предыдущих спринтов, на 2026-04-24 известно:

```
datasets/
├── sputnik8/cards.raw.jsonl      # 30 карточек
├── pmpoperator/cards.raw.jsonl   # 14 карточек
├── scantour/cards.raw.jsonl      # 14 карточек
├── afisha/cards.raw.jsonl        #  9 карточек
└── schema/
    ├── product_card.schema.json
    └── test_case.schema.json
```

Каждая строка `cards.raw.jsonl` — объект вида:

```json
{
  "card": { /* product_card */ },
  "_meta": {
    "source_site": "sputnik8",
    "source_url": "https://www.sputnik8.com/ru/...",
    "fetched_at": "...",
    "parser_version": "v1",
    "json_ld_found": false,
    "warnings": []
  }
}
```

Проверь счёт в начале сессии: `wc -l datasets/*/cards.raw.jsonl` должно
дать `67` в сумме. Если число расходится — сперва расспроси
пользователя, а не размечай.

`text_rules.yaml` — 35 текстовых правил, каждое с `id` (TXT-NN),
`category`, `title`, `description`, `example`, `severity` из {`low`,
`medium`, `high`, `critical`}.

## Выход

### 1. Файлы кейсов

Разложить по двум каталогам:

```
datasets/cases/
├── real-clean/
│   └── <case_id>.json          # expected_clean: true, violations: []
└── real-dirty/
    └── <case_id>.json          # expected_clean: false, violations: [≥1]
```

**Один файл = один кейс = одна исходная карточка.** Формат — валидный
`card_case` из `test_case.schema.json`:

```json
{
  "case_id": "sputnik8_24725",
  "kind": "card_case",
  "source": "production",
  "generator": {
    "model": null,
    "prompt_version": null,
    "date": "2026-04-25"
  },
  "card": { /* один-в-один card из cards.raw.jsonl */ },
  "expected_violations": [
    {
      "rule_id": "TXT-19",
      "severity": "medium",
      "field_path": "full_description",
      "quote": "лучший тур в Карелии, гарантированно",
      "rationale": "Сравнительное утверждение без основания — TXT-19 запрещает необоснованные суперлативы."
    }
  ],
  "expected_clean": false,
  "notes": null
}
```

**Ключевые правила формата:**

- `case_id` = `card.id` исходной карточки (уже уникален в рамках
  датасета).
- Имя файла: `<case_id>.json`.
- `source` = `"production"` (всегда).
- `generator.date` — сегодняшняя дата в `YYYY-MM-DD`.
- `generator.model` / `generator.prompt_version` — `null` (реальная
  выгрузка, не синтетика).
- В `expected_violations[].severity` — **ровно та** severity, что у
  правила в `text_rules.yaml` / `image_rules.yaml`. Расхождение =
  ошибка; быстрая проверка — вспомогательный скрипт (см. ниже).
- `quote` — **дословный** фрагмент из исходного поля (не перефраз). Для
  TXT-правил `quote` обязателен (антигаллюцинационный якорь).
- `field_path` — дот-нотация до поля, где лежит нарушение. Примеры:
  - `title`
  - `short_description`
  - `full_description`
  - `program_items[2].description`
  - `services[0].name`
  - `contacts_block.public_comment`
  - `location.meeting_point_comment`
- `rationale` — 1–2 предложения, почему именно это правило и почему
  именно этот фрагмент.
- `notes`:
  - `null` если разметка уверенная.
  - `"review"` если агент сомневается → поднять с пользователем
    отдельным сообщением по итогу спринта (не блокируя работу).

**Схема-инвариант** (проверяется `pnpm parse:validate` после доработки
валидатора, либо разовым скриптом):

- `expected_clean: true` ⇒ `expected_violations = []`.
- `expected_clean: false` ⇒ `expected_violations.length ≥ 1`.

Отдельный короткий раздел в финальном ответе:
- Сколько карточек помечено `notes: "review"` и по каким конкретно
  фрагментам нужен человек. Список — в одно сообщение, без файлов.

## Процесс

### Предпосылки

1. **Прочитай `text_rules.yaml` целиком** (для IMG-разметки —
   `image_rules.yaml`). Таксономия — единственный источник правды.
   Пример, описание и severity правила — ориентир.
2. **Прочитай `datasets/schema/test_case.schema.json` → `card_case`.**
   Это форма выхода. Поля `rule_id`, `severity`, `field_path`, `quote`,
   `rationale` — обязательные внутри `violation_card`.
3. **Не правь парсеры и `text_rules.yaml` / `image_rules.yaml`.** Мусор
   в карточке — это сигнал, а не баг: возможно, именно такие данные
   попадают в боевую модерацию.

### Пошагово

Для каждой из 67 карточек:

1. **Field sanity-check.**
   - Поля не пустые там, где их быть не должно (title, full_description,
     images[0]).
   - Нет артефактов парсера (HTML-теги в plain-text, mojibake, обрывки
     JSON).
   - Если явный мусор (парсер поймал нерелевантный блок) — `notes:
     "review"`, причину фиксируем человеческим языком и **двигаемся
     дальше**. Не считаем это нарушением `text_rules.yaml` /
     `image_rules.yaml`.

2. **Прогон по 35 TXT-правилам.**
   - Пройди по каждому TXT-01…TXT-35 по очереди (не по категориям —
     именно по ID, чтобы ничего не пропустить).
   - Для каждого правила:
     - Смотри его `description` и `example` в `text_rules.yaml`.
     - Проверь ВСЕ текстовые поля карточки: `title`,
       `short_description`, `full_description`, каждый
       `program_items[i].title/description`,
       `services[i].name/description`, `location.*`,
       `contacts_block.public_comment`.
     - Если нашёл нарушение — зафиксируй (`rule_id`, `severity` из
       правила, `field_path`, дословный `quote`, `rationale`).
   - Одна карточка может нарушать несколько правил; одно правило может
     срабатывать на несколько полей — каждое такое срабатывание =
     отдельный элемент `expected_violations[]`.

3. **Классификация и запись файла.**
   - Нарушений 0 → `expected_clean: true`, `expected_violations: []` →
     `datasets/cases/real-clean/<card.id>.json`.
   - Нарушений ≥ 1 → `expected_clean: false` → `real-dirty/<card.id>.json`.
   - Любое сомнение → `notes: "review"` и продолжаем: массовый прогон
     важнее идеальной разметки 1–2 кейсов.

4. **Запись в аудит-отчёт.**
   - Инкрементно обновляй счётчики clean/dirty/review по источнику.
   - Веди счёт по rule_id, чтобы финальная таблица собралась без
     отдельного прохода.

### Что такое «реальное нарушение» (калибровка)

**Порог**: применяй правило так, как его применил бы модератор
Localtrip — не «по возможному смыслу», а «очевидно нарушает».

Примеры того, что **не** стоит размечать как нарушение (избегаем
false-positives):

- Стилистическая неаккуратность без содержательного нарушения.
- Стандартные маркетинговые обороты («увлекательная экскурсия») без
  конкретных ложных обещаний.
- Короткое `full_description` — это слабое наполнение, не нарушение.
- Отсутствие полей (парсер не нашёл `schedule.dates`) — это не
  нарушение карточки, а пробел парсинга.

Примеры того, что **нужно** размечать:

- Телефон / ссылка на мессенджер в `full_description` — TXT-07.
- «Пишите напрямую — сделаем без комиссии» — TXT-06.
- Фактические несоответствия между `title` и `program_items`
  («5-дневный тур», а программа — 1 день) — TXT-21.
- Явная дискриминация / оскорбления — TXT-10, TXT-11.
- Гарантии, которых нет в продуктовой логике («гарантия возврата в
  любой момент») — TXT-03.

Если сомневаешься между «слабовато» и «тянет на нарушение» —
`notes: "review"` с конкретным вопросом.

### Проверка формата перед коммитом

Прогнать одноразовую валидацию: все файлы в `datasets/cases/real-*/*.json`
парсятся как JSON, соответствуют `card_case`, и `expected_clean` ↔
`expected_violations.length` консистентны.

Если валидатора на `test_case.schema.json` ещё нет, добавить
лёгкий скрипт `scripts/validate-cases.ts` (не обязательно pnpm-команда —
можно просто tsx). Один экран, ajv как в `lib/schema.ts`.

## Hard rules

1. **Не редактируй `text_rules.yaml` / `image_rules.yaml`.** Даже если
   видишь, что правило можно уточнить — вопрос пользователю, не правка.
2. **Не редактируй `cards.raw.jsonl`.** Источник иммутабельный. Если
   карточка непригодна, помечай в `notes: "review"`.
3. **Не удаляй `datasets/<source>/cards.raw.jsonl`** — они пригодятся
   следующим спринтам. Только добавляй в `datasets/cases/real-*/`.
4. **severity** в кейсе = severity в правиле. Никаких даунгрейдов
   «потому что тут мягко».
5. **quote — дословно.** Копипаст из поля; не «перефраз того, что
   сказано». Пустая строка / отсутствие `quote` для TXT-правила ⇒
   кейс невалидный.
6. **Никаких персданных в `rationale`/`notes`.** Если в карточке был
   телефон/почта, в `quote` он останется (это и есть нарушение TXT-07),
   но в `rationale` не копируй.
7. **Не смешивай clean+dirty.** Если нарушений нет — строго в
   `real-clean/`, даже если карточка «так себе по качеству».

- [ ] Для каждого `violation` с `rule_id` типа `TXT-*` поле `quote`

## Открытые вопросы для начала сессии

1. **Разметка через рабочую Promptfoo-обёртку или сыро?**
   Рекомендую сыро — `card_case` уже спроектирован под Promptfoo,
   обёртку собирать не нужно.
2. **Что делать, если карточка очевидно «сломана парсером»** (например,
   `full_description` содержит обрывки JS)? Рекомендую: `notes:
   "review"`, классифицировать всё равно (по полям, которые в порядке),
   но пометить — по итогу спринта пользователь решит, доразмечать или
   пересобрать парсер.
3. **Длина rationale.** 1–2 предложения — для всех правил, даже если
   нарушение «очевидно». Цель — объяснить модели в будущем, почему
   ground truth именно такой.

Ответы — в начале сессии, до того как агент начнёт массово размечать.
