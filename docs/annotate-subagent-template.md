# Шаблон делегации субагенту на разметку

Короткий промпт-шаблон для запуска субагента (Task / Agent /
inline-делегация) на разметку pending-карточек. Канонические инструкции
не дублируются — субагент читает их по ссылкам ниже. Это снижает дрейф
между параллельными разметками и экономит токены (200-строчный промпт →
~30 строк делегации + переиспользуемые артефакты).

## Когда использовать

- Партия pending'ов одного источника (≥3 карточек) — массовая разметка.
- Точечный пересмотр конкретного `case_id` после изменения правил.
- Параллельные субагенты на независимые подмножества (по 5–10 карточек
  на агента, чтобы не превышать его контекст).

Для одиночной разметки одной карточки в текущей сессии шаблон
избыточен — открой pending руками или скорми его LLM напрямую с
`prompts/annotate-conservative-v1.txt`.

## Шаблон

```markdown
Задача: разметить pending-файлы из <source> по правилам.
Режим: conservative (по умолчанию) | aggressive (только во втором проходе).

## Канонические артефакты — читать в первую очередь

- `prompts/annotate-conservative-v1.txt` — основной промпт
  (для второго прохода — `prompts/annotate-aggressive-v1.txt`).
- `datasets/text_rules.compact.json` — таблица TXT-правил для
  текстовой разметки.
  (Для image-разметки — `datasets/image_rules.compact.json`.)
- `docs/annotation-guide.md` — workflow, формат вывода, частые ошибки.
- `datasets/schema/annotation.schema.json` — JSON Schema разметки.

## Файлы для разметки

<список case_id или путь к pending-директории>

## Специфика источника <source>

<2–3 пункта характерных нарушений по опыту>:
- <напр. «sputnik8: длинные программы, фокус на TXT-19/TXT-20 — обещания»>
- <напр. «afisha: мало текста, чаще формальные нарушения наполнения»>

## Что вернуть

- Перезаписать каждый pending-файл (Write) — заполнить
  `expected_clean`, `violations`, `annotator`, `annotated_at`.
  Поле `annotator` — твой model-id (см. инструкцию в промпте).
- В финальном сообщении — компактная сводка:
  `total / clean / dirty / нарушения по rule_id`.
- НЕ вызывать `pnpm annotations:commit` — это шаг основного агента
  после ревью.
```

## Двухпроходная разметка

Если запускается двухпроходная схема (см. соответствующий раздел
`docs/annotation-guide.md`), субагент получает два независимых вызова
шаблона: один с conservative-промптом → результаты в
`pending/<case_id>.cons.json`, второй с aggressive-промптом → в
`pending/<case_id>.aggr.json`. Merge и фильтрация — отдельная задача
основного агента, не субагента.

## Что НЕ должен делать субагент

- Не править `text_rules.yaml` / `image_rules.yaml` и compact-таблицы —
  таксономия только под явный запрос пользователя (см. `CLAUDE.md`).
- Не запускать `pnpm annotations:commit` / `cases:generate` —
  материализация в store и cases — шаг основного агента после ревью.
- Не выходить за `rule_id` из compact-таблицы — своих идентификаторов
  не выдумывать.
